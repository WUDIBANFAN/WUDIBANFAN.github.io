<!--more-->

## 一、什么是驾驭工程（Harness Engineering）

**Harness Engineering**（驾驭工程）是一套系统化的 AI 编程方法论，核心理念是：**把 AI 当作一个能自主完成复杂开发任务的智能体，而非一个被动响应指令的聊天工具。** 它的关键在于为 AI 提供足够清晰、结构化、可验证的上下文约束，让 AI 能够自主规划、执行、验证并交付完整的开发成果。

与之相对的是 **Prompt Engineering**（提示词工程），后者的典型工作模式是：人不断发现 bug → 告诉 AI → AI 修 bug → 人再发现 bug → 再告诉 AI……如此反复。这种"chat 式修 bug"的流程不仅效率低下，而且让开发者始终无法从项目中抽身。

驾驭工程追求的是：**一次性把事情做对。** 如果 AI 的产出结果不理想，我们首先应该反思是不是没有跟 AI 交代清楚，而不是靠不断聊天来补救。这就是驾驭工程与提示词工程最本质的区别。

## 二、驾驭工程的三件"马具"

驾驭工程的三个核心组件分别是：地图（agents.md）、目标（spec 规格文档）和验收标准（acceptance criteria）。下面逐一详细拆解。

### 2.1 第一件马具：地图 —— `agents.md`

在项目的根目录下创建一个名为 `agents.md`（或 `CLAUDE.md`、`CURSOR_RULES.md`，取决于所用工具）的文件，这就是 AI 的项目地图。它的作用是让 AI 在一开始就全面了解这个项目的整体情况，而不需要自己去"探索"。

虽然现在的 AI 已经足够聪明，不给任何信息它也能自己探索并了解项目，但探索过程会：

- **大量消耗 token**（即浪费费用和时间）
- **分散注意力**，AI 可能在无关的方向上浪费时间
- **容易遗漏关键信息**，导致后续返工

所以，关键信息一开始就要主动交给 AI。

#### `agents.md` 必须包含的信息

**（1）项目一句话描述**

用一句话把项目是做什么的说清楚。例如：

> 本项目使用 Godot 引擎开发一个 2D 横版卷轴小游戏，主题是"地牢探险"。

这一句话本质上定义了 AI 的工作边界——凡是超出这个范围的，AI 就不会去做了。

**（2）项目仓库结构说明**

清楚说明代码、资源文件、文档分别对应哪些文件夹。例如：

```
项目仓库结构：
- src/              —— 游戏源代码（GDScript 脚本）
- assets/           —— 美术资源（精灵图、音效、字体）
- scenes/           —— Godot 场景文件（.tscn）
- docs/             —— 项目文档
- specs/            —— 规格文档
- tests/            —— 测试脚本
```

这让 AI 知道去哪里找东西、去哪里放东西，不会把代码写到文档目录里。

**（3）开发约束（按需补充）**

也就是明确"不能做什么"和"必须遵守哪些规则"。例如：

- 必须使用 GDScript 1.0（不使用 C# 版本）
- 不引入第三方插件，只用 Godot 内置节点
- 所有文本内容使用中文
- 不考虑移动端适配，目标平台是桌面端（Windows/macOS）

**重要提醒**：`agents.md` 是**目录而非说明书**，不要把所有细节都塞进来。它的作用是导航，不是替代 spec 文档。保持简洁、结构化，让 AI 能快速定位到关键信息。

---

### 2.2 第二件马具：目标 —— 规格文档（Spec）

规格文档一般放在 `specs/`（或 `spec/`）文件夹下面，内容要写得**非常详细**。如果说 `agents.md` 是地图，那么规格文档就是目的地和路线规划。

一份好的规格文档应包含以下内容：

**（1）项目背景与愿景**

项目的终极目标是什么？解决了什么问题？目标用户是谁？

**（2）核心机制（Game Mechanics / Core Features）**

如果是游戏项目，要详细描述：

- 玩家操作方式（移动、跳跃、攻击等）
- 关卡结构（关卡数量、难度递进）
- 得分/生命系统
- 敌人 AI 行为逻辑
- 道具/升级系统
- 胜负判定条件

如果是非游戏项目，则详细描述核心功能逻辑、业务流程、状态流转等。

**（3）视觉风格与 UI 规范**

- 整体美术风格方向（像素风、扁平化、拟物化等）
- 配色方案（主色、辅色、强调色）
- 字体与字号规范
- UI 布局原则
- 动画/过渡效果

**（4）版本交付内容**

明确这个版本具体要交付哪些内容，并按优先级排列：

- P0（必须有）：可玩的核心游戏循环
- P1（应该有）：基础 UI、音效
- P2（可以有）：额外关卡、特效增强

#### 拆分策略：主文档 + 子文档

如果要写的内容很多，可以拆分成两层结构：

```
specs/
├── main-spec.md           —— 主规格文档（总览）
├── game-mechanics.md      —— 子文档：游戏机制
└── visual-standards.md    —— 子文档：视觉校验
```

在 `main-spec.md` 中用链接引用子文档，例如：

```markdown
详细游戏机制见 [game-mechanics.md](./game-mechanics.md)。
视觉风格规范见 [visual-standards.md](./visual-standards.md)。
```

这种分层的好处是：AI 可以按需读取子文档，而不是一股脑把所有信息都塞进上下文窗口。

### 2.3 第三件马具：验收标准（Acceptance Criteria）

验收标准是驾驭工程中最容易被忽视但最关键的一环。它必须满足以下原则：

#### （1）具体、能量化

能给出具体数字的地方，绝不模糊。例如：

- ❌ 错误写法："敌人不能太强也不能太弱"
- ✅ 正确写法："玩家初始攻击力为 10，第一关普通敌人的生命值为 30，确保玩家 3 次攻击击杀一个敌人"

#### （2）明确判定方法

对于可能"蒙混过关"的地方，一定要明确写出判定方法。例如：

- ❌ 错误写法："游戏运行流畅"
- ✅ 正确写法："在 60FPS 目标下，整个游戏过程中的帧率不得低于 55FPS，使用 Godot 内置的 Performance 面板截图作为证据"

#### （3）要求 AI 提供测试通过的证据

**在源头上杜绝 AI 偷懒。** 每一条验收标准后面都要要求 AI 提供具体的证据，例如：

- 截图证据：关键界面的截图
- 日志证据：运行日志中的关键输出
- 测试报告：自动化测试的执行结果
- 视频/GIF：动态效果的录屏

例如一条完整的验收标准可以这样写：

> **AC-1：玩家移动系统**
>
> - 玩家按 A/D 键可以左右移动，移动速度为 200px/s
> - 按空格键可以跳跃，跳跃高度为 150px
> - **证据要求**：提供一段 5 秒的录屏，展示玩家左右移动和跳跃

## 三、AI 不能既当运动员又当裁判

以上三件马具搭建好之后，Harness Engineering 的基本框架就完成了。但这里藏着一个致命的问题：

**AI 既写代码又当测试，最后还要验收自己开发的成果。**

如果直接让 AI 开干，你会发现不到十几分钟，AI 就告诉你"任务已完成"，但实际产出的质量却一塌糊涂——因为 AI 在测试和验收环节会不自觉地"放水"，跳过边界情况、忽略非功能性需求、自圆其说地通过所有验收标准。

**核心原则：不能让 AI 既是运动员又是裁判，必须让一群 AI 分工合作、互相制衡。**

### 3.1 多代理协作（Multi-Agent Collaboration）

解决方案是将开发流程拆分成不同角色，交给不同的子代理（Sub-agents）来执行：

| 角色             | 职责                                         |
| ---------------- | -------------------------------------------- |
| 规划代理         | 解读规格文档，制定开发计划，拆分任务         |
| 开发代理         | 根据任务清单编写代码                         |
| 测试代理         | 根据验收标准编写测试用例，执行测试           |
| Code Review 代理 | 审查代码质量，检查最佳实践                   |
| 验收代理         | 对照验收标准逐条检查，收集证据，生成验收报告 |

分工之后，最明显的变化是：

1. **开发时长会显著拉长**——从十几分钟变成数小时。这不是坏事，说明 AI 在认真做事。
2. **最终产出的质量会大幅提升**——因为每个环节都有专门的 AI 把关，不存在"自己给自己放水"的问题。

### 3.2 实现方式一：手动编排（Superpowers 插件）

手动创建子代理并编排它们的工作是一种方法，但比较繁琐。更推荐的入门方案是使用 **Superpowers** 插件。

Superpowers 是一个开源的工作流插件，它能自动创建多个子代理，并安排好任务和分工。安装方法很简单：

1. 在 Claude Code（或支持 MCP 的 AI 编程工具）中打开插件面板
2. 在搜索栏输入 `superpowers`
3. 点击添加即可完成安装

不同的代理工具安装方式可能会有所不同，可参考其官方文档：
[GitHub](https://github.com/obra/superpowers)

装好之后的用法：

1. 点击加号创建新的对话
2. 选择插件，然后选中 Superpowers
3. 其他操作和普通对话一样，直接告诉 AI 要做什么

不同的是，这一次 AI 会自动创建多个子代理，每个子代理都会明确自己的职责。例如：

- 负责写测试用例的子代理会根据验收标准设计详详细的测试用例
- 代码开发则会交给另外的子代理来实现

各个子代理之间还会互相通信、传递信息，形成一个完整的工作流水线。

### 3.3 实现方式二：手动创建子代理

手动创建子代理是一种更精细的控制方式，适合复杂项目。基本思路是：

1. 为每个子代理编写独立的系统提示词（System Prompt），定义其角色和职责
2. 规划子代理之间的通信协议（谁把结果传给谁）
3. 编写编排脚本，按依赖关系串行或并行调度子代理

这种方法需要在未来单独展开详细讲解。

## 四、实操演示：用 Superpowers 开发一个 Godot 2D 小游戏

下面以 "使用 Godot 开发一个 2D 地牢探险小游戏" 为例，展示完整的驾驭工程实操流程。

### 步骤 1：编写 `agents.md`

在项目根目录创建 `agents.md`（完整内容如下）：

```markdown
# 项目地图

## 项目概述
使用 Godot 4.x 引擎开发一个 2D 地牢探险小游戏。

## 仓库结构
- src/          —— 游戏源代码（GDScript）
- assets/       —— 美术资源（精灵图、音效）
- scenes/       —— Godot 场景文件
- specs/        —— 规格文档
- tests/        —— 测试脚本

## 开发约束
- 必须使用 GDScript，不使用 C#
- 不依赖第三方插件，仅使用 Godot 内置节点
- 目标平台：桌面端（Windows/macOS/Linux）
- 所有游戏内文本使用中文
```

### 步骤 2：编写规格文档

在 `specs/main-spec.md` 中写：

```markdown
# Godot 2D 地牢探险小游戏 - 主规格文档

## 项目愿景
打造一个轻量级的 2D 地牢探险游戏，玩家控制角色在地牢中探索、战斗、收集道具，最终击败关底 Boss。

## 版本范围（v0.1）
本版本的交付范围：
- P0：玩家移动、攻击、生命值系统
- P0：一个可游玩的地牢关卡（含 5 种敌人 + 1 个 Boss）
- P1：道具收集系统
- P1：基础 UI（血条、分数）

## 分文档链接
- 游戏机制详见：[game-mechanics.md](./game-mechanics.md)
- 视觉风格详见：[visual-standards.md](./visual-standards.md)
```

然后分别在 `game-mechanics.md` 和 `visual-standards.md` 中展开详细内容。

### 步骤 3：编写验收标准

在 `specs/acceptance-criteria.md` 中写：

```markdown
# 验收标准

## AC-1：玩家移动系统
- 按 A/D 或方向键左右移动，速度 200px/s（±5% 误差允许）
- 按空格跳跃，跳跃高度 150px（±5% 误差允许）
- 证据：提供 5 秒录屏，展示左右移动 + 跳跃

## AC-2：玩家攻击系统
- 按 J 键攻击，攻击范围 50px，伤害 10
- 攻击动画持续 0.3s，期间玩家不可移动
- 证据：提供录屏展示攻击动画 + Debug 信息面板显示的伤害数值

## AC-3：成就系统完整覆盖
- 详细参照 specs/game-mechanics.md 中 12 项成就与解锁条件，全部通过验证
- 证据：测试报告列出 12 项成就对应测试用例的执行结果

...（其余验收标准略）
```

### 步骤 4：启动 Superpowers 执行

1. 安装 Superpowers 插件（在插件面板搜索 "superpowers"，点击添加）
2. 新建对话 → 选择插件 → 选中 Superpowers
3. 输入指令：

> 请按照 specs/ 目录下的规格文档和验收标准，使用 Godot 引擎开发完整的 2D 地牢探险小游戏。请参考 agents.md 了解项目约束。所有子代理必须按照验收标准逐条验证，并提供通过证据。

4. AI 会自动创建多个子代理（规划代理、开发代理、测试代理、验收代理等），分工协作，持续工作数小时，最终交付完整的游戏代码和验收报告。

## 五、驾驭工程 vs 提示词工程：终极对比

| 维度     | 提示词工程（Prompt Engineering） | 驾驭工程（Harness Engineering）        |
| -------- | -------------------------------- | -------------------------------------- |
| 核心思路 | 不断和 AI 对话，通过聊天修正 bug | 一次性提供充分约束，让 AI 自主完成     |
| 交互方式 | 人机多轮对话（几十轮甚至上百轮） | 少轮次交互，AI 长时间自主工作          |
| 文档要求 | 随意的自然语言描述               | 结构化的地图 + 规格 + 验收标准         |
| 验证方式 | 人手动测试，发现问题再反馈       | AI 自动验证，提供证据报告              |
| 分工模式 | 一个人 + 一个 AI                 | 多个 AI 子代理分工协作                 |
| 开发时长 | 不可控，取决于"发现-反馈"的轮数  | 较长但可预测，AI 自主执行              |
| 产出质量 | 不稳定，依赖人的发现和反馈质量   | 稳定可控，有明确的验收标准兜底         |
| 适合场景 | 小改动、临时需求、探索性编码     | 完整项目、复杂功能、需要交付成果的场景 |

## 六、总结与核心要点

驾驭工程的关键在于：**清晰地定义目标、约束和验收标准，再交给不同的子代理去分工执行，最后由独立的代理进行验收。**

三大核心要素：

1. **地图（agents.md）**：项目导航，告诉 AI 去哪找东西、项目边界在哪
2. **目标（specs）**：详细规格，让 AI 清楚地知道要建造什么
3. **验收标准（acceptance criteria）**：可量化、可验证、需要提供证据，杜绝 AI 偷懒

一个核心原则：**不能让 AI 既当运动员又当裁判。必须让多个 AI 代理分工协作、互相制衡。**

一句判断标准：**如果 AI 的产出结果不理想，先反思是不是没有跟 AI 交代清楚，而不是靠聊天来补救。** 这就是驾驭工程与提示词工程最本质的区别。
