## 一、RAG 基础流水线

### 1.1 标准检索流程

完整链路如下：

```
文档分块 → 向量化入库（向量库）+ 倒排索引入库（BM25）
                                              ↓
用户 Query → 多路粗排检索（向量相似度 + BM25 关键词）→ 合并候选文档 → Rerank 重排序（精排）→ 截取 Top-N → 拼接 Prompt → LLM 生成答案
```

### 1.2 粗排 vs 精排

| 维度 | 粗排（向量/BM25） | 精排（Rerank） |
|------|-------------------|----------------|
| **目标** | 高召回，尽可能捞回潜在相关文档 | 高精度，对少量候选做细粒度打分 |
| **速度** | 极快，百万级向量库毫秒返回 | 较慢，仅作用于几十条候选 |
| **返回量** | Top 10/20/50 | Top 3~8 |
| **缺陷** | 打分逻辑简单，容易混入噪声文档 | 不适合全库遍历，成本较高 |

> 粗排牺牲精度换召回，精排牺牲速度换精度，二者级联配合是工业界主流方案。

## 二、Rerank 精排：为什么需要它？

### 2.1 粗排的固有缺陷

1. **向量相似度的语义错位**
   - 向量检索是全局向量距离匹配，存在"字面相近但语义无关"的问题。
   - 例：Query 问"电动车续航"，文档大量出现"电池"向量相近，但实际内容讲储能电站，和车辆无关——向量打分很高但无效。

2. **BM25 的关键词盲区**
   - BM25 仅匹配关键词词频，无法理解语义逻辑。
   - 例：Query"怎么清洗汽车座椅"，文档高频出现"座椅、清洗"但讲的是办公椅，BM25 分数靠前。

3. **多路打分标准不一致**
   - 向量余弦分和 BM25 分数值域完全不同，直接融合加权容易失衡。

### 2.2 Rerank 的核心作用

1. **细粒度语义精准打分**
   - Rerank 模型（Cross-Bert、BGE-Reranker、MiniRerank 等）采用 **Query + 文档拼接输入**，做交叉注意力计算，直接判断二者语义相关性，区分"字面相似"与"真实相关"。

2. **统一多路检索打分标准**
   - 不管是向量召回还是 BM25 召回的文档，全部用同一个 Rerank 模型输出 0~1 标准化相关分，公平排序。

3. **过滤粗排噪声，提升 Precision@K**
   - 粗排为了高召回会带入大量低相关文档，Rerank 把无关文档压到列表末尾，送入 LLM 的上下文全部是高价值参考文本，减少幻觉、提升回答准确度。

4. **降低 LLM 上下文负担**
   - 粗排返回 50 条候选，Rerank 筛选 Top 3~8 送入 LLM，减少 Prompt 长度、降低 token 消耗、加速 LLM 推理。

### 2.3 Rerank 落地知识点

- **轻量化方案**：不用大参数量 Cross-Bert，采用蒸馏后的小 Rerank 模型（如 `bge-reranker-v2-m3`）平衡速度与精度。
- **阈值截断**：Rerank 打分低于设定阈值（如 0.3）直接丢弃，不送入 LLM。
- **成本权衡**：候选集过大（>100 条）时 Rerank 耗时陡增，因此粗排返回候选数量要做限制，是 Top-K 动态调整的联动依据。

## 三、检索指标：Recall@K 与 Precision@K

### 3.1 基础定义

给定 Query，标准答案相关文档集合为 `Rel`，检索返回 Top-K 文档集合为 `Ret`。

| 指标 | 公式 | 含义 | 关注点 |
|------|------|------|--------|
| **Recall@K** | $\frac{\|Rel \cap Ret\|}{\|Rel\|}$ | 所有相关文档中，有多少被找出来了 | 不漏信息 |
| **Precision@K** | $\frac{\|Rel \cap Ret\|}{\|Ret\|}$ | 返回的 K 条中，真正相关的占比 | 不掺垃圾 |

### 3.2 天然矛盾

- **提高 Recall**：需要扩大检索范围、增大 Top-K、放宽阈值 → 必然混入无关文档 → Precision 下降
- **提高 Precision**：缩小检索范围、减少返回条数、严格过滤 → 容易漏掉部分相关文档 → Recall 下降

二者此消彼长，无法同时无限拉高，必须按业务场景取舍。

### 3.3 不同业务场景取舍策略

#### 场景 1：高召回优先（Recall@K 优先）

| 项目 | 内容 |
|------|------|
| **适用业务** | 知识库问答、企业内部文档检索、法律/医疗专业问答 |
| **核心诉求** | 绝对不能漏掉关键信息，漏关键文档会导致 LLM 回答错误 |
| **优化手段** | 粗排 Top-K 拉大（30~50 条）；开启 BM25+向量混合检索；Rerank 仅做排序，不做强阈值丢弃 |
| **代价** | Precision 降低，依赖 Rerank 后续过滤噪声 |

#### 场景 2：高精度优先（Precision@K 优先）

| 项目 | 内容 |
|------|------|
| **适用业务** | C 端简短问答、客服快速问答、短文本查询 |
| **核心诉求** | LLM 上下文窗口有限，不允许冗余文本占用 token，噪声会引发幻觉 |
| **优化手段** | 缩小粗排 Top-K（10 条以内）；Rerank 设置严格分数阈值；向量相似度设下限提前过滤 |
| **代价** | 少量小众相关文档可能无法被召回，Recall 轻微下降 |

#### 场景 3：平衡方案（工业界通用最优解）

绝大多数 RAG 系统采用**高召回粗排 + 高精度 Rerank 精排**组合：

1. 粗排阶段牺牲 Precision，拉满 Recall，保证不漏任何潜在有用文档
2. Rerank 阶段做精准筛选，过滤噪声，最终送入 LLM 的少量文档保证高 Precision

兼顾两端指标，是项目落地最常用的取舍思路。

### 3.4 延伸评价指标

- **F1@K**：调和 Recall 与 Precision，用于离线模型调参统一评估。
- **业务最终指标**：不只用检索指标，端到端看 LLM 回答正确率、幻觉率，检索指标只是中间过程指标。

## 四、Top-K 动态调整

### 4.1 静态 Top-K 的缺陷

固定返回 K 条存在明显短板：

1. **简单短 Query**（如"公司上班时间"）仅需要 1 条文档，固定返回 20 条浪费算力、拉长 LLM 上下文。
2. **复杂长 Query**（如"2025 年全年财务报销流程及各类发票要求"）需要多条文档，固定返回 5 条会丢失信息，Recall 暴跌。
3. **不同知识库文档密度不同**，部分主题资料多、部分资料少，统一 K 值无法适配。

### 4.2 动态调整四大方案

#### 方案 1：基于 Query 复杂度动态分配 K 值

- 离线训练 Query 分类器或用大模型 Prompt 分类，区分简单短 Query / 复杂多意图 Query。
- 规则：
  - 单意图、短问句 → K=3~5
  - 多意图、长问题、专业复杂查询 → K=20~50
- 轻量实现：统计 Query 长度、关键词数量做简易划分，无需训练模型。

#### 方案 2：基于相似度分数阈值自适应截断（最常用）

粗排检索时，不强制固定 K，而是持续返回文档直到满足两个条件之一：

1. 达到最大上限 $K_{max}$（如 50 条，防止无限返回）
2. 文档向量相似度低于全局阈值（如 0.6），后续文档全部丢弃

> 逻辑：资料充足的主题会提前截断，资料稀缺主题拉满 $K_{max}$ 兜底召回。

#### 方案 3：基于 Rerank 分数动态筛选

粗排返回固定大候选集（如 30 条），经过 Rerank 打分后：

1. 保留分数高于阈值的全部文档
2. 同时设置最大上限（最多 8 条送入 LLM，控制上下文长度）

> 示例：若只有 2 条文档打分达标，就只传 2 条；若 10 条达标，只截取 Top 8。

#### 方案 4：基于知识库文档密度动态调节

离线统计每个文档分块主题下的文档数量：

- 小众主题、文档稀少 → 调大 K
- 热门主题、文档海量 → 调小 K，减少冗余

### 4.3 工程落地约束

1. **上限保护**：必须设置 $K_{max}$，防止极端低质 Query 触发全库检索，压垮向量库服务。
2. **Token 约束联动**：动态 K 不能无视 LLM 上下文窗口，汇总文档总 token 超过限制时，强制截断低 Rerank 分文档。
3. **性能监控**：动态 K 越大，Rerank 耗时越高，线上需监控检索耗时，高延迟场景自动下调 K 值做降级。

## 五、BM25 与向量检索互补

### 5.1 底层原理差异

| 维度 | BM25（词法检索） | 向量检索（语义检索） |
|------|-------------------|----------------------|
| **核心原理** | 基于词频、逆文档频率、文档长度匹配关键词 | 通过 Embedding 将文本映射到高维向量空间 |
| **语义理解** | 无，只识别字面词汇 | 有，捕获深层语义 |
| **优势** | 专有名词、术语、编号、ID 精确匹配极强 | 同义词、改写、转述、口语化适配好 |
| **短板** | 同义词、转述、语义改写完全失效 | 专有名词、冷僻术语、短编号容易 Embedding 偏移 |
| **训练成本** | 无 | 依赖训练数据，小语种/垂直领域容易效果滑坡 |

> 例：文档写"新能源汽车续航里程"，用户问"电车能跑多远"，无重合关键词，BM25 完全召回不到，但向量检索可以精准召回。

### 5.2 为什么 BM25 效果好时仍需向量检索？

#### 理由 1：用户查询存在大量语义改写

企业内部、C 端用户很少使用和文档完全一致的专业关键词，大量口语化、转述提问，仅靠 BM25 会严重丢失相关文档，Recall 暴跌。

> 例：制度文档写"员工年度带薪休假申请流程"，用户提问"年假怎么申请"，BM25 关键词匹配弱，向量检索可以精准召回。

#### 理由 2：混合召回互补，大幅提升整体 Recall

BM25 擅长精准专有名词匹配，向量擅长语义泛化，两路召回取并集，能覆盖单一检索遗漏的文档。

| 检索方式 | Recall@20 |
|----------|-----------|
| 仅 BM25 | ~72% |
| 仅向量 | ~76% |
| **混合两路召回** | **~91%** |

即便单独 BM25 指标好看，混合后依然有显著增益。

#### 理由 3：垂直领域长尾 Query 依赖向量泛化

知识库存在大量长尾问题，文档不会穷举所有提问方式，纯关键词检索无法覆盖，向量语义匹配是兜底方案。

### 5.3 BM25 效果极好时的轻量化折中

如果业务场景 90% Query 都是精确术语查询，BM25 已经足够，可对向量检索做轻量化降低成本：

1. **多路召回开关**：简单精确 Query 仅走 BM25，复杂语义 Query 才触发向量检索。
2. **缩小向量召回 Top-K**：向量仅返回少量候选，降低向量库计算压力。
3. **轻量 Embedding 模型**：选用小参数量向量化模型，减少编码耗时。

### 5.4 混合检索融合策略

两路召回文档合并后打分融合方案：

1. **加权融合**：BM25 分数、向量余弦分归一化后加权求和。
2. **全部送入 Rerank 统一重排**（工业界主流，无需手动调权重，交给模型判断相关性）。

## 六、RAG 优化全链路总结

回答"你的 RAG 做过哪些优化"时，可按以下逻辑串联：

1. **召回层优化**：BM25 + 向量多路混合检索，互补词法 + 语义；动态 Top-K 自适应调整，平衡 Recall 与算力。
2. **精排层优化**：引入 Rerank 交叉编码器做细粒度语义排序，过滤粗排噪声，提升 Precision。
3. **指标调优**：根据业务场景取舍 Recall@K 与 Precision@K，采用"高召回粗排 + 高精度精排"平衡方案。
4. **工程落地优化**：轻量化模型、阈值截断、上下文 token 限流、检索链路降级策略。

## 七、常见面试反问预判

**Q：Rerank 带来额外耗时怎么解决？**

轻量化蒸馏 Rerank 模型、限制粗排候选数量、高并发场景做 Rerank 服务批量推理、简单 Query 跳过 Rerank 直接取 BM25 结果。

**Q：混合检索权重怎么调？**

优先用 Rerank 统一打分，避免手动加权；离线数据集网格搜索权重，以端到端 LLM 正确率为最终评估标准。

**Q：如何离线评估 Recall/Precision？**

构建 Query-标注相关文档测试集，批量跑检索链路，计算各 K 值下指标，指导 Top-K、Rerank 阈值调参。
