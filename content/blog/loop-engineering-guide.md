开一个定时任务让 AI 自己循环干活——触发、执行、验收，三步闭环，人在睡觉，AI 在写代码。

<!--more-->

## 一、什么是循环工程（Loop Engineering）

**Loop Engineering**（循环工程）是驾驭工程（Harness Engineering）的进阶形态。如果说驾驭工程是"给 AI 套上马具让它跑一趟"，那循环工程就是"给 AI 装上一个自动驾驶的巡航系统，让它到了路口自己判断要不要接着跑下一趟"。

它的核心机制是三个步骤的闭环：

```
触发条件 → 处理任务 → 验收结果 →（未通过）→ 再次触发 → ...
                                ↘（通过）→ 循环结束
```

循环工程让开发者可以真正实现"人在睡觉，AI 在干活"——但它对任务的清晰度和验收标准的要求比驾驭工程更加苛刻，因为一旦出错，后果不是一段烂代码，而是无限循环烧 token，让你一觉醒来直接掉入"斩杀线"。

## 二、循环工程的三个步骤

### 步骤 1：明确触发条件 —— 什么时候有活干

触发条件是循环的起点，告诉 AI "什么时候该醒过来干活"。触发条件必须是**可被程序化检测的事件**，常见的触发条件包括：

- **新工单（Ticket）到达**：比如 Jira、Linear、GitHub Issues 中来了一个新的 bug 工单或功能需求
- **新的 Pull Request（PR）被提交**：当团队成员提交了新的 PR，AI 自动进行 Code Review
- **定时任务触发**：每天凌晨 2 点，AI 自动检查依赖库是否有安全更新，如有则生成升级 PR
- **监控告警触发**：生产环境报错率超过阈值，AI 自动分析日志、定位问题、生成修复方案
- **邮件到达**：收到特定格式的催活邮件（如"请处理 XXX 需求"），AI 解析邮件内容后开始执行

关键原则：**触发条件必须是明确、可检测、不会产生歧义的信号。** 不能是"感觉该干活了"或"当项目需要改进的时候"这种模糊描述。

### 步骤 2：处理任务 —— 根据触发条件执行对应动作

触发条件决定了 AI 要执行什么任务。不同的触发条件对应不同的工作流：

| 触发条件       | AI 的处理任务                                                |
| -------------- | ------------------------------------------------------------ |
| 新 bug 工单    | 分析 bug 描述 → 定位代码 → 编写修复 → 生成测试 → 提交 PR |
| 新功能需求工单 | 阅读 spec → 编写功能代码 → 编写测试 → 更新文档 → 提交 PR |
| 新 PR 提交     | 阅读 diff → 检查代码风格 → 运行测试 → 生成 Review 意见    |
| 定时安全扫描   | 拉取依赖列表 → 查询 CVE 数据库 → 生成升级 PR               |
| 监控告警       | 拉取日志 → 分析错误模式 → 定位根因 → 生成修复方案         |

**"处理任务"这一步的 AI 只负责执行，不负责判断自己做得对不对。**

### 步骤 3：验收结果 —— 决定循环是否结束（最关键的一步）

**这是整个循环工程中最关键的一步。** 如果验收环节出错，可能导致两种严重后果：

- **验收太松**：AI 交付了错误的代码，但被判定为"通过"，bug 代码直接合入主分支
- **验收太严（或逻辑错误）**：AI 永远无法通过验收，陷入无限循环，持续烧 token，一夜之间费用爆表

因此，验收标准必须具备以下特征：

1. **可量化、可自动化验证**：不能靠"看起来不错"来判断，必须有具体的、可程序化检查的指标
2. **有明确的终止条件**：如果重试 N 次仍不通过，必须终止循环并通知人类介入
3. **有 timeout 机制**：单次循环的最长执行时间必须有上限

## 三、循环工程最常见的两个错误

很多人一开始会这样理解循环工程——写一段提示词交给定时任务跑：

> "检查有没有工单，有的话处理掉工单，别搞错，不然老子弄死你。"

这犯了循环工程中最致命的两个错误：

### 错误一：目标模糊

"处理掉工单"——什么叫处理掉？是把工单标记为"已完成"就算处理掉，还是必须修完 bug、写完测试、通过了 CI 才算处理掉？

"别搞错"——什么算搞错？是没有语法错误就算对，还是所有测试通过、边缘情况覆盖了才算对？

**模糊的目标 = 不可验证的结果 = 不可控的循环。**

正确的写法应该是：

> 检查 GitHub Issues 中 label 为 `bug` 且 assignee 为 `ai-bot` 的工单。如果存在此类工单，则：
>
> 1. 阅读工单中描述的 bug
> 2. 在 `src/` 目录下找到相关代码
> 3. 编写修复代码
> 4. 编写对应的单元测试
> 5. 运行 `npm test`，确保所有测试（含新增测试）通过
> 6. 生成 PR，标题格式为 `fix: [issue-#ID] 工单标题`
> 7. 循环结束条件：CI 流水线全部绿色 + 测试覆盖率不低于变更前的水平
> 8. 最多重试 3 次，3 次后若仍未通过则发送 Slack 通知给 @developer

### 错误二：同一个 AI 既当裁判又当运动员

这是循环工程中最重要的原则，也是从驾驭工程一脉相承的核心铁律：

> **干活的 AI 和验收的 AI 必须是不同的。**

如果一个 AI 既写代码又判断自己写的代码对不对，它几乎一定会"放水"——跳过边界条件、忽略非功能性需求、在验收标准上自我妥协。这就是所谓的"既当运动员又当裁判"。

正确做法是：

- **Worker Agent（干活代理）**：负责步骤 2——接收任务、执行开发
- **Verifier Agent（验收代理）**：负责步骤 3——独立验证 Worker 的产出，对照验收标准逐条检查，决定循环是否结束
- **Orchestrator（编排器）**：负责步骤 1——监听触发条件，调度 Worker 和 Verifier，管理循环状态

三者各司其职，互相制衡。

## 四、循环工程的六大元素

Google 工程师 **Andy Osmani** 总结了循环工程的六个核心元素，每个元素都有对应的工具支持：

### 1. 自动化（Automation）

循环的"发动机"。负责定时触发、监听事件、启动工作流。

- **Claude Code 的 Loop 功能**：内置的循环执行能力
- **Codex 的自动化**：支持定时任务和事件触发
- **GitHub Actions / GitLab CI**：传统的 CI/CD 流水线，可以加入 AI 步骤
- **Cron Jobs**：最基础的定时触发器

### 2. 工作流（Workflows）

定义"触发之后要做什么"的流程模板。一个工作流是一个有向无环图（DAG），节点是任务，边是依赖关系。

例如一个"新 bug 工单"的工作流：

```
[收到工单] → [分析bug] → [定位代码] → [编写修复]
                                              ↓
                                         [编写测试]
                                              ↓
                                         [运行测试] → [通过?]
                                              ↓            ↓
                                             是           否
                                              ↓            ↓
                                         [提交PR]    [通知人类]
                                              ↓
                                         [验收代理验证]
                                              ↓
                                    [通过?] → [结束循环]
                                        ↓
                                       否 → [Worker 重试]
```

### 3. 技能（Skills）

每个 AI 代理所具备的专项能力。不同的任务需要不同的技能组合：

- 编写代码（不同语言、不同框架）
- 编写测试（单元测试、集成测试、E2E 测试）
- Code Review（风格检查、安全审查、性能分析）
- 文档撰写（API 文档、Changelog、迁移指南）
- 日志分析（错误模式识别、根因定位）

### 4. 连接器（Connectors）

AI 与外部系统之间的桥梁。没有连接器，AI 无法感知外部世界。常见的连接器包括：

- **GitHub/GitLab API**：读取 Issues、PR，提交代码
- **Slack/Discord 通知**：发送告警、汇报进度
- **邮件系统（IMAP/SMTP）**：读取工单邮件、发送回复
- **监控系统（Datadog/Prometheus）**：读取告警、拉取日志
- **CI/CD 系统**：触发流水线、读取运行结果

### 5. 子代理（Sub-agents）

循环工程中的分工执行者。如上文所述，至少需要两种子代理：

- **Worker Agent**：执行具体任务
- **Verifier Agent**：独立验收

在复杂场景下，还可以细分出更多角色：规划代理、测试代理、文档代理、安全审查代理等。

### 6. 记忆（Memory）

**说白了就是写文档。** 循环工程的记忆层负责：

- **记录循环状态**：当前循环是第几次重试、上一次失败的原因是什么
- **持久化上下文**：让不同循环之间能共享信息（上一个循环修了什么 bug，这个循环不要重复修）
- **积累项目知识**：每次循环的经验沉淀为文档，提高后续循环的效率

记忆的实现方式就是前面驾驭工程中提到的 `agents.md`、`specs/` 等结构化文档，以及 AI 循环执行过程中自动更新的日志和状态文件。

## 六、循环工程的适用边界

循环工程不是万能的。它有一个明确的能力边界：

### 适合循环工程的任务

- **目标清晰、可量化、可自动化验证**的任务
- 有明确的触发条件
- 有既定的处理流程和验收标准
- 不需要创造性思维

典型场景：

- 自动修 bug（工单驱动）
- 自动 Code Review（PR 驱动）
- 自动生成 Release Notes（定时驱动）
- 自动依赖升级（定时驱动）
- 自动回复标准格式的客户邮件（邮件驱动）

### 不适合循环工程的任务

- 目标模糊的任务：**"给我一个 amazing 的创意"**
- 没有客观评判标准的任务：**"帮我搞钱"**
- 需要大量不确定探索的任务：**"研究一下这个新技术方向"**
- 需要人类审美和主观判断的任务：**"设计一个好看的 UI"**

**一句话总结：循环工程只能执行目标清晰的任务。如果你要 AI 干的事是"给我一个 amazing 的创意"或者"帮我搞钱"——那臣妾做不到。**

## 七、循环工程 vs 驾驭工程 vs 提示词工程：三层对比

| 维度       | 提示词工程             | 驾驭工程                 | 循环工程                            |
| ---------- | ---------------------- | ------------------------ | ----------------------------------- |
| 触发方式   | 人手动输入             | 人手动输入一次           | 自动化事件触发                      |
| 执行次数   | 单次对话               | 单次长时任务             | 持续循环执行                        |
| 人的参与度 | 全程参与（多轮对话）   | 一次设定，等待结果       | 几乎零参与                          |
| 验收方式   | 人肉眼检查             | AI 子代理自验收          | 独立 AI 代理验收 + 终止保护         |
| 失败模式   | 代码不对，人发现后反馈 | 结果不完整，人检查后返工 | 无限循环烧 token / 错误代码静默合入 |
| 风险控制   | 人实时把关             | 事后检查                 | 必须内置重试上限和超时机制          |
| 适合场景   | 小改动、探索           | 完整项目、复杂功能       | 持续运营、重复性维护任务            |

## 八、总结与核心要点

循环工程的核心机制：**开一个定时/事件驱动的任务，让 AI 按照"触发 → 执行 → 验收"的闭环自动干活。**

六大元素：**自动化、工作流、技能、连接器、子代理、记忆。**

两大核心原则：

1. **目标必须清晰可验证**——模糊的目标必然导致不可控的结果
2. **干活的 AI 和验收的 AI 必须是不同的**——不能让 AI 既当裁判又当运动员

一道红线：**只做目标清晰的任务。** 创意类、探索类、主观评判类的任务不适合循环工程。
